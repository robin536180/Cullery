# Findings

## 当前结论

- 项目已经从纯 Web Demo 演进到可向移动端落地的 MVP 形态，核心基座是 `React + Vite + Tailwind + Capacitor + PWA`。
- 中英文切换能力已接入，语言资源位于 `src/i18n/locales/en.json` 与 `src/i18n/locales/zh.json`，初始化位于 `src/i18n/config.ts`。
- 设置页与语音页已经接入原生能力封装，当前通过 `src/services/native.ts` 提供权限检查、权限申请、模型下载模拟等接口。
- Android 原生工程已经初始化完成，`npx cap sync` 成功，Web 构建产物已同步到 `android/app/src/main/assets/public/`。
- 当前“相册权限、麦克风权限、模型下载”在 MVP 中仍以演示/模拟为主，并未真正完成全链路原生落地。

## 已确认的问题

- 真实手机相册读取尚未实现，目前 `NativeService.getPhotos()` 仍返回空数组，业务数据仍主要依赖前端 mock 数据。
- 大模型自动下载当前是进度模拟，并未真正使用文件下载、断点续传、存储校验、模型解压和落盘逻辑。
- 语音能力当前仅完成权限申请与交互演示，未形成真实的 ASR 识别结果到业务指令执行的完整闭环。
- Android Debug APK 在当前环境中未能完成构建，原因是 Gradle 依赖下载被网络限制拦截，报错为 `Connection refused`。
- Web 产物体积偏大，`vite build` 结果中主包约 `847.63 kB`，后续需要做拆包与懒加载优化。

## 风险判断

- 如果下一阶段直接追求“手机端真实体验”，优先级最高的不是 UI，而是原生相册接入、真实语音识别、模型下载与本地存储。
- 真正落地 7B 模型到手机端时，包体、下载时长、磁盘占用、权限说明和设备兼容性会成为首批核心风险。
- Android 虚拟机测试完全可行，适合验证安装流程、权限流程、页面逻辑与基础原生桥接，但不适合验证真实相册规模、机型性能和大模型推理体验。

## 建议

- 先完成“可安装 + 可授权 + 可读取真实相册 + 可执行基础清理”的真机链路，再推进本地大模型。
- 先用轻量规则/小模型跑通相册分析与清理工作流，再把 7B 模型作为增强能力异步接入。

## 跟踪建议

- 每次推进真实能力时，优先记录“是否已脱离 mock”。
- 每个关键能力建议同时记录“Web 演示状态”和“Android 真机状态”。
- 从现在开始，建议把“是否可在安卓虚拟机完成演示”作为阶段验收门槛之一。
