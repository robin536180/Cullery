# 系统架构设计：效率优先的四层引擎

整个系统设计围绕“快速扫描、精准识别、安全删除”的流水线优化。

text

用户照片库（RAW/HEIC/JPEG）

↓

[引擎层1：元数据与快速过滤器]

基于时间/地点/相册的快速聚类

基于哈希的**完全重复文件**识别  $\rightarrow$  直接安全删除

调用系统API识别**截图、录屏、文档照片**  $\rightarrow$  建议归档或删除

↓

[引擎层2：视觉相似度聚类]

使用轻量模型（如MobileViT）提取图像特征向量

对特征进行高效聚类，找出“相似组”（如连拍、同一场景多角度）

↓

[引擎层3：多维度智能仲裁]

输入：一个“相似组”照片（通常2-10张）

仲裁流程：

1. **技术筛选**：使用轻量IQA模型剔除严重模糊、闭眼等废片。

2. **专业评分**：对剩余照片，并行运行多个“微模型”进行评分：

- 画质模型（清晰度、曝光）

- 构图模型

- 场景/时刻模型（黄金时刻、稀有背景）

- 情感模型（表情、互动）

3. **加权决策**：根据场景类型应用不同权重，计算出每张照片的“综合保留价值分”。

4. **输出**：推荐保留1-3张（根据场景），其余明确标记为“可安全删除”。

核心：所有计算在设备本地完成，无需网络。

↓

[引擎层4：安全删除与整理]

- 将“可删除”照片移入应用内的**“回收站”**，而非系统相册回收站。

用户可一键确认删除，或二次浏览。

被保留的照片，可自动打上智能标签（人物、地点、事件），便于后续查找。

# 2. 移动端开发框架

- 首选：Flutter

- 理由：一套代码覆盖iOS和Android，UI性能优异，开发效率高。对于复杂的自定义相机和图像处理，可通过Platform Channel调用原生模块实现。

o 图像处理：使用C++编写高性能的图像解码、缩放、特征提取流水线，通过FFI供Flutter调用。

- 模型推理：iOS使用Core ML，Android使用TFLite Native API，以获得最佳性能。

# 3.存储与隐私架构

- 本地数据库：使用SQLite，存储照片特征向量、聚类结果、用户决策记录。

- 隐私设计：

。所有流程在Sandbox内完成，照片数据绝不外传。

。权限最小化：可选"仅整理最近一个月照片"模式。

。数据可解释性：展示AI决策的简单依据（如“删除原因：模糊”）。
